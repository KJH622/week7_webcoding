## 1. 실제로 사용하는 서비스

### 1-1. DB 엔진 (가장 직접적인 사용)

| 서비스 | 사용 방식 |
|---|---|
| MySQL (InnoDB) | 테이블 데이터 자체를 B+ 트리로 저장. `CREATE INDEX` 시 자동으로 B+ 트리 생성 |
| PostgreSQL | 기본 인덱스 타입 = B+ 트리. `CREATE INDEX` 명시 없으면 자동 적용 |
| SQLite | 테이블 저장 + 인덱스 전부 B+ 트리 |
| Oracle DB | 기본 인덱스 타입 = B 트리 |
| CouchDB | 문서 + 뷰 인덱스 모두 B+ 트리. MVCC + append-only 설계 결합 |
| MongoDB | WiredTiger 스토리지 엔진 내부적으로 B 트리 계열 사용 |

---

### 1-2. 소셜 미디어 / 콘텐츠

#### 인스타그램
- **DB**: PostgreSQL (B+ 트리 기본 인덱스)
- B+ 트리 성능을 극대화하기 위해 **ID 구조 자체를 설계**한 케이스
- UUID 대신 **64비트 정수** 사용:
  - 41비트: 타임스탬프 (밀리초)
  - 13비트: 샤드 ID
  - 10비트: 자동 증가 시퀀스

#### 트위터 (X), 디스코드
- 인스타그램과 동일한 이유로 순차적 64비트 정수 ID 채택
- UUID처럼 랜덤한 값은 B+ 트리에서 노드 분할을 자주 유발 → 성능 저하

> **참고**: 랜덤 UUID를 B+ 트리에 삽입하면 순차 정수 대비 삽입 속도가 **10배 ~ 100배 느려질 수 있음**

---

### 1-3. 이커머스 / 결제

#### 에어비앤비
- MySQL 대규모 운영, B+ 트리 인덱스 최적화를 핵심 엔지니어링 과제로 다룸

#### 쿠팡, 배달의민족, 당근마켓
- MySQL / PostgreSQL 기반
- 주문 ID, 상품 ID, 사용자 ID 인덱스가 전부 B+ 트리로 동작

---

### 1-4. 모빌리티 / O2O

#### 우버 (Uber)
- 초기 PostgreSQL → MySQL(InnoDB)로 전환한 유명한 사례
- **전환의 핵심 원인이 B 트리 인덱스 문제**:
  - B 트리는 주기적으로 리밸런싱이 필요하고, 이 과정에서 서브트리가 통째로 새 위치로 이동
  - 대규모 **쓰기 증폭(Write Amplification)** 발생
  - 인덱스 12개짜리 테이블에서 컬럼 하나 업데이트 시 12개 인덱스를 모두 갱신하는 오버헤드
- B+ 트리 구조가 서비스 아키텍처 설계에 얼마나 큰 영향을 주는지 보여주는 대표 사례

---

### 1-5. 브라우저 / 모바일 기기

#### 크롬, 사파리
- 브라우저 히스토리, 북마크, 쿠키 저장에 **SQLite** 내부적으로 사용
- SQLite = B+ 트리 기반

#### 카카오톡, 인스타그램 앱, 갤러리, 연락처 앱
- 앱 내부 로컬 DB가 거의 전부 SQLite → B+ 트리
- 채팅 내역, 사진 메타데이터, 연락처가 B+ 트리로 저장 / 조회됨

---

### 1-6. 파일 시스템

| OS / 파일 시스템 | 사용 방식 |
|---|---|
| macOS (HFS+, APFS) | 파일 / 디렉토리 메타데이터 인덱싱에 B 트리 계열 사용 |
| Linux (ext4, Btrfs) | 디렉토리 및 메타데이터 관리에 B 트리 / B+ 트리 변형 사용 |
| Windows (NTFS) | 파일 시스템 메타데이터 관리에 B 트리 계열 사용 |

---

### 1-7. SSD 하드웨어

- SSD 내부 **플래시 변환 계층(FTL)** 의 매핑 테이블도 B 트리 계열 구조 사용
- 파일을 저장할 때 SSD 칩 내부에서도 B 트리가 동작하고 있음

## 2. 무조건 B+ 트리가 빠른가?

**아니다. 상황에 따라 다르다.**

### B 트리가 더 빠른 경우

**단일 키 탐색 (Best Case)**
- B 트리는 내부 노드에도 데이터가 있어서, 운이 좋으면 루트나 중간 노드에서 바로 찾고 끝냄
- B+ 트리는 항상 리프까지 내려가야 함

```
찾는 값이 루트에 있으면:
  B 트리  → 1번 비교로 완료
  B+ 트리 → 무조건 리프까지 내려가야 함
```

### B+ 트리가 더 빠른 경우

**범위 탐색 (`BETWEEN`, `>`, `<`)**
- 리프 노드가 연결 리스트로 이어져 있어서 시작점만 찾으면 옆으로 쭉 읽으면 됨
- B 트리는 범위 탐색 시 트리를 다시 올라갔다 내려갔다 해야 해서 훨씬 느림

**대용량 순차 읽기**
- 전체 데이터를 순서대로 읽는 경우 리프 연결 리스트 덕분에 압도적으로 빠름

**디스크 기반 대용량 DB**
- 내부 노드에 데이터가 없으니 노드 하나에 더 많은 키를 저장 가능
- 트리 높이가 낮아짐 → 디스크 I/O 횟수 감소

### 요약 비교표

| 상황 | 더 빠른 쪽 |
|---|---|
| 단일 탐색 (운 좋은 경우) | B 트리 |
| 단일 탐색 (평균 / 최악) | 비슷함 |
| 범위 탐색 | **B+ 트리** |
| 전체 순회 | **B+ 트리** |
| 디스크 기반 대용량 DB | **B+ 트리** |

## 3. 3방향 벤치마크 아이디어

### 방향

다른 조와 차별점을 위해 **선형 탐색 / B 트리 / B+ 트리 세 가지를 동시에 비교**하는 벤치마크 구성

### 벤치마크 구성

| 탐색 종류 | 단일 탐색 (`id = ?`) | 범위 탐색 (`BETWEEN`) |
|---|---|---|
| 선형 탐색 | 측정 | 측정 |
| B 트리 | 측정 | 측정 |
| B+ 트리 | 측정 | 측정 |

레코드 수별로 (10만 / 50만 / 100만 / 500만) 반복 측정

### 기대 효과

- "B+ 트리를 선택한 이유"를 데이터로 증명 가능
- 범위 탐색에서 B+ 트리가 압도적으로 유리한 것을 숫자로 보여줄 수 있음
- 발표 시 "우리 팀은 세 가지 방식을 직접 구현하고 비교했다"는 설득력 확보

### 주의사항

- B+ 트리 완성 우선 → 여유 생기면 B 트리 추가하는 순서로 진행
- B 트리 구현이 벅차면 과감하게 빼는 것이 나음
- "B 트리도 구현했는데 미완성" 보다 "B+ 트리 하나를 완벽하게"가 훨씬 나음

## 4. 프로젝트 컨셉 — GameRank DB

### 컨셉 선정 이유

- 100만 유저라는 규모가 현실적으로 납득됨
- 플레이어 ID로 점수를 조회하는 구조가 `WHERE id = ?` 과제 요구사항과 1:1로 맞아떨어짐
- 범위 탐색 (`점수 BETWEEN 9000 AND 10000`) 도 자연스럽게 설명 가능
- 발표할 때 흥미를 끌기 좋음

### 테이블 스키마

```c
typedef struct {
    int  id;        // Primary Key (자동 부여)
    char name[32];  // 플레이어 닉네임
    int  score;     // 점수 (0 ~ 9999)
    char tier[16];  // 티어 (Challenger / Diamond / Platinum / Gold / Silver)
} Player;
```

### 티어 기준

| 티어 | 점수 기준 |
|---|---|
| Challenger | 9500 이상 |
| Diamond | 8500 이상 |
| Platinum | 7000 이상 |
| Gold | 5000 이상 |
| Silver | 5000 미만 |

## 5. UI 구성 방향

### 전체 레이아웃

```
┌─────────────────────────────────────────┐
│  GameRank DB       B+ Tree Index Demo   │
│  [총 플레이어 수] [트리 높이] [마지막 탐색] │  ← 요약 카드 3개
├─────────────────────────────────────────┤
│  Player ID 입력창          [레코드 수 선택] │
│  [단일 탐색] [범위 탐색] [TOP 10 랭킹]     │  ← 탐색 유형 선택
│  [ 탐색 실행 버튼 ]                        │
├─────────────────────────────────────────┤
│  선형 탐색   │   B 트리   │   B+ 트리 ★   │  ← 3개 카드 (순서대로 결과 채워짐)
│  시간 / 횟수 │ 시간 / 횟수 │  시간 / 횟수  │
│  [막대 게이지]│[막대 게이지]│ [막대 게이지] │
├─────────────────────────────────────────┤
│  랭킹 테이블 (검색한 플레이어 하이라이트)    │
└─────────────────────────────────────────┘
```

### 차별점 포인트

- 탐색 유형을 **단일 / 범위 / TOP 10** 으로 전환하면 B+ 트리가 압도적으로 유리한 경우가 시각적으로 보임
- 탐색 실행 시 세 카드가 **순서대로 결과를 채우는 애니메이션** → 발표 데모로 활용 가능
- 실제 C 구현에서 측정한 숫자를 그대로 넣으면 발표용 데모로 사용 가능
