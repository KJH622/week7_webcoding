## 1. 실제로 사용하는 서비스

### 1-1. DB 엔진

| 서비스 | 사용 방식 |
|---|---|
| MySQL (InnoDB) | 테이블 데이터 자체를 B+ 트리로 저장. `CREATE INDEX` 시 자동으로 B+ 트리 생성 |
| PostgreSQL | 기본 인덱스 타입 = B+ 트리. `CREATE INDEX` 명시 없으면 자동 적용 |
| SQLite | 테이블 저장 + 인덱스 전부 B+ 트리 |
| Oracle DB | 기본 인덱스 타입 = B 트리 |
| CouchDB | 문서 + 뷰 인덱스 모두 B+ 트리. MVCC + append-only 설계 결합 |
| MongoDB | WiredTiger 스토리지 엔진 내부적으로 B 트리 계열 사용 |

### 1-2. 소셜 미디어

#### 인스타그램
- **DB**: PostgreSQL (B+ 트리 기본 인덱스)
- B+ 트리 성능을 극대화하기 위해 **ID 구조 자체를 설계**한 케이스
- UUID 대신 **64비트 정수** 사용:
  - 41비트: 타임스탬프 (밀리초)
  - 13비트: 샤드 ID
  - 10비트: 자동 증가 시퀀스

#### 트위터 (X), 디스코드
- 동일한 이유로 순차적 64비트 정수 ID 채택
- UUID처럼 랜덤한 값은 B+ 트리에서 노드 분할을 자주 유발 → 삽입 속도 **10~100배** 저하

### 1-3. 이커머스 / 결제

| 서비스 | 내용 |
|---|---|
| 에어비앤비 | MySQL 대규모 운영, B+ 트리 인덱스 최적화를 핵심 엔지니어링 과제로 다룸 |
| 쿠팡, 배달의민족, 당근마켓 | MySQL / PostgreSQL 기반. 주문·상품·사용자 ID 인덱스 전부 B+ 트리로 동작 |

### 1-4. 모빌리티

#### 우버 (Uber)
- 초기 PostgreSQL → MySQL(InnoDB)로 전환한 유명한 사례
- 전환의 핵심 원인이 **B 트리 인덱스 문제**:
  - B 트리 리밸런싱 시 서브트리가 통째로 새 위치로 이동 → 대규모 **쓰기 증폭(Write Amplification)**
  - 인덱스 12개짜리 테이블에서 컬럼 하나 업데이트 시 12개 인덱스를 모두 갱신해야 하는 오버헤드

### 1-5. 브라우저 / 모바일 앱

| 서비스 | 내용 |
|---|---|
| 크롬, 사파리 | 히스토리, 북마크, 쿠키 저장 → SQLite(B+ 트리) 내부 사용 |
| 카카오톡, 갤러리, 연락처 앱 | 앱 내부 로컬 DB가 거의 전부 SQLite → B+ 트리 |

### 1-6. 파일 시스템 / 하드웨어

| 시스템 | 내용 |
|---|---|
| macOS (HFS+, APFS) | 파일 / 디렉토리 메타데이터 인덱싱에 B 트리 계열 사용 |
| Linux (ext4, Btrfs) | 디렉토리 및 메타데이터 관리에 B 트리 / B+ 트리 변형 사용 |
| Windows (NTFS) | 파일 시스템 메타데이터 관리에 B 트리 계열 사용 |
| SSD 하드웨어 | 플래시 변환 계층(FTL) 매핑 테이블에 B 트리 계열 구조 사용 |

## 2. 유명 서비스는 어떤 DBMS를 통해 B+ 트리를 사용하는가?

대부분의 서비스는 B+ 트리를 **직접 구현하지 않고 DBMS가 제공하는 인덱스를 통해** 사용한다.

```
서비스 코드
    ↓  SQL 쿼리 (CREATE INDEX, SELECT WHERE)
DBMS (MySQL InnoDB / PostgreSQL)
    ↓  내부적으로
B+ 트리 인덱스 자동 관리
```

| 서비스 | DBMS | B+ 트리 사용 방식 |
|---|---|---|
| 인스타그램, 에어비앤비 | PostgreSQL | `CREATE INDEX` → 자동으로 B+ 트리 생성 |
| 우버, 트위터 | MySQL InnoDB | Primary Key 자체가 B+ 트리 클러스터드 인덱스 |
| 카카오톡 앱 내부 | SQLite | 테이블 자체가 B+ 트리 구조로 저장 |
| 넷플릭스, 쿠팡 | MySQL / PostgreSQL | 사용자 ID, 콘텐츠 ID 등 조회 빈도 높은 컬럼에 인덱스 적용 |

## 3. 왜 대규모 서비스는 WHERE id = ? 조회에 인덱스를 반드시 거는가?

**인덱스 없이 100만 건 테이블에서 하나를 찾으면 최악의 경우 100만 번 비교가 필요하다.**

실측 데이터 예시 (PostgreSQL 5백만 건 기준):

| 방법 | 실행 시간 |
|---|---|
| 인덱스 없는 풀스캔 | 173 ms |
| B+ 트리 인덱스 사용 | 0.07 ms |
| 성능 차이 | **약 2,500배** |

대규모 서비스에서 이 차이가 중요한 이유:
- 동시에 수천 명의 유저가 요청을 날리면 173ms × 1,000 = 초당 수십 초의 누적 지연
- DB 서버 CPU와 I/O가 풀스캔으로 포화되면 다른 쿼리들도 덩달아 느려짐
- 인덱스가 없으면 데이터가 늘어날수록 응답 속도가 **선형으로 느려지지만**, 인덱스가 있으면 **거의 일정하게 유지**됨

## 4. B 트리보다 B+ 트리가 범위 검색에 유리한 이유

핵심 차이: **리프 노드 연결 리스트**

```
B 트리 범위 탐색 (BETWEEN 30 AND 60):
  루트 → 내부 → 30 찾기 → 다시 루트로 올라감 → 내부 → 40 찾기 → ... (반복)
  매 값마다 트리를 처음부터 다시 탐색

B+ 트리 범위 탐색:
  루트 → 내부 → 리프(30) 찾기 → 옆 리프(40) → 옆 리프(50) → 옆 리프(60) → 종료
  시작점 한 번만 찾으면 연결 리스트를 따라 쭉 읽으면 됨
```

| 탐색 방식 | B 트리 | B+ 트리 |
|---|---|---|
| 단일 탐색 (id = ?) | O(log n), 운 좋으면 중간에서 찾음 | O(log n), 항상 리프까지 내려감 |
| 범위 탐색 (BETWEEN) | O(k × log n) — k번 반복 탐색 | O(log n + k) — 한 번 탐색 후 연결 리스트 순회 |
| 전체 순회 | 트리 전체 순회 필요 | 리프 연결 리스트만 순회 |

> `ORDER BY id`, `BETWEEN`, `>`, `<` 조건이 많은 실제 서비스에서 B+ 트리가 압도적으로 유리한 이유다.

## 5. 풀스캔과 인덱스 탐색의 체감 성능 차이

실제 서비스 사례:

**소셜 미디어 (게시글 피드 조회)**
- 200만 건 posts 테이블, `created_at` 컬럼 인덱스 없음
- 피드 로딩 시간: **8초** → 인덱스 추가 후 **수십 ms**

**이커머스 (주문 조회)**
- 고객 주문 내역 조회 쿼리
- 인덱스 없음: **10초** → 인덱스 추가 후 **수 ms**

**결론**: 데이터가 적을 때는 풀스캔도 빠르지만, 수십만 건을 넘어서면 인덱스 없는 쿼리는 **사용자가 체감할 수 있는 수준으로 느려진다**.

예외적으로 풀스캔이 유리한 경우:
- 테이블이 매우 작을 때 (수백 건 이하)
- 결과가 전체 데이터의 30% 이상을 반환하는 쿼리 (이 경우 옵티마이저가 풀스캔을 선택하기도 함)

## 6. 리프 노드 연결 구조가 서비스 성능에 주는 이점

B+ 트리의 리프 노드는 **이중 연결 리스트**로 연결되어 있다.

```
[리프1: 10,13] → [리프2: 15,22] → [리프3: 30,40] → [리프4: 50,55] → ...
```

이 구조가 주는 이점:

**1. 범위 탐색 (BETWEEN)**
- 시작 리프만 찾으면 끝까지 옆으로 이동하면서 읽기만 하면 됨
- 디스크 I/O 관점에서 **순차 읽기**가 가능해 속도가 빠름

**2. ORDER BY 최적화**
- 이미 정렬된 순서로 연결되어 있어서 별도 정렬 작업 불필요
- 랭킹 조회, 최신순 피드 등에서 큰 이점

**3. 전체 데이터 순회**
- 루트부터 탐색할 필요 없이 맨 왼쪽 리프에서 연결 리스트만 따라가면 전체 스캔 가능

**실제 적용 예시**:
- 게임 랭킹 TOP 100 조회 → 리프 연결 리스트 순회로 처리
- 소셜 피드 시간순 정렬 → ORDER BY created_at 시 리프 연결 리스트 활용

## 7. 실제 DBMS 인덱스 vs 우리 과제의 메모리 기반 B+ 트리

| 비교 항목 | 실제 DBMS (MySQL InnoDB) | 우리 과제 (메모리 기반) |
|---|---|---|
| 저장 위치 | 디스크 (페이지 단위 16KB) | RAM (포인터 기반) |
| 노드 크기 | 고정 페이지 크기 (16KB) | 구조체 크기 (수십 바이트) |
| 핵심 알고리즘 | **동일** (삽입, 분할, 리프 연결 리스트) | **동일** |
| 트리 높이 | 실제 100만 건 = 약 3~4 레벨 | 동일 |
| 영속성 | 디스크에 저장, 서버 재시작 후에도 유지 | 프로세스 종료 시 사라짐 |
| 동시성 제어 | 트랜잭션, 잠금(Lock), MVCC | 없음 (단일 스레드) |
| 버퍼 풀 | 자주 쓰는 노드를 메모리에 캐싱 | 전부 메모리 (캐싱 불필요) |
| 성능 병목 | 디스크 I/O | 없음 (메모리 접근만) |

**같은 것**: 트리 탐색 로직, 노드 분할 규칙, 리프 연결 리스트 구조, O(log n) 시간 복잡도

**다른 것**: 디스크 기반 vs 메모리 기반, 트랜잭션/동시성 처리 여부

> 이번 과제는 "핵심 알고리즘"을 구현하는 것이라 실제 DBMS와 본질은 동일하다.

## 8. 왜 id 검색은 인덱스를 타고 name/score는 선형 탐색이 될 수 있는가?

**인덱스는 특정 컬럼에만 만들어지기 때문이다.**

```sql
-- id 컬럼에 인덱스 있음 → B+ 트리 탐색
SELECT * FROM players WHERE id = 500000;     -- O(log n) ✓

-- name 컬럼에 인덱스 없음 → 풀스캔
SELECT * FROM players WHERE name = 'Alice';  -- O(n) ✗

-- score 컬럼에 인덱스 없음 → 풀스캔
SELECT * FROM players WHERE score > 9000;    -- O(n) ✗
```

**왜 모든 컬럼에 인덱스를 안 거는가?**
- 인덱스를 만들 때마다 별도 B+ 트리 자료구조가 생겨 **저장 공간 증가**
- INSERT / UPDATE / DELETE 시 모든 인덱스를 갱신해야 해서 **쓰기 속도 저하**
- 실무에서는 자주 조회하는 컬럼에만 선택적으로 인덱스를 건다

**이번 과제에서의 적용**:
- `id` 컬럼만 B+ 트리 인덱스 대상
- `name`, `score` 조건은 선형 탐색 → 이 차이를 벤치마크로 비교하는 것이 과제의 핵심

## 9. 삽입이 많을 때도 왜 트리 인덱스를 유지하는가?

삽입 시 인덱스 유지 비용:
- 리프 노드에 키 삽입
- 노드가 꽉 차면 **분할(Split)** 발생 → 부모 노드에 키 추가
- 최악의 경우 루트까지 분할이 전파됨

그럼에도 인덱스를 유지하는 이유:

**1. 읽기가 쓰기보다 압도적으로 많다**
- 대부분의 서비스에서 읽기 : 쓰기 비율은 **10:1 ~ 100:1**
- 쓰기 때 약간 느려지더라도 읽기가 수천 배 빠르면 전체적으로 이득

**2. 분할 비용은 O(log n)으로 제한된다**
- 분할이 루트까지 전파되는 최악의 경우도 O(log n) 수준
- 100만 건 기준 약 20번 안에 처리 완료

**3. 대안인 LSM Tree도 같은 문제를 다른 방식으로 해결**
- 카산드라, RocksDB처럼 쓰기가 극단적으로 많으면 LSM Tree를 선택
- 일반적인 서비스는 B+ 트리가 더 적합

## 10. B+ 트리 vs 해시 인덱스 vs LSM Tree — 언제 각각 유리한가?

### 세 가지 비교

| 구분 | B+ 트리 | 해시 인덱스 | LSM Tree |
|---|---|---|---|
| 단일 탐색 (id = ?) | O(log n) | **O(1)** | O(log n) |
| 범위 탐색 (BETWEEN) | **최강** | 불가 | 가능하지만 느림 |
| 정렬 (ORDER BY) | **최강** (이미 정렬됨) | 불가 | 가능 |
| 쓰기 성능 | 보통 | 보통 | **최강** |
| 읽기 성능 | **최강** | 단일 탐색만 빠름 | 보통 |
| 사용 DB | MySQL, PostgreSQL | MySQL MEMORY 엔진 | Cassandra, RocksDB |

### 언제 각각 선택하는가?

**B+ 트리를 선택하는 경우**
- 범위 탐색, 정렬, 단일 탐색이 모두 필요한 일반적인 서비스
- 모르겠으면 B+ 트리가 기본값

**해시 인덱스를 선택하는 경우**
- `WHERE id = ?` 같은 단순 동등 비교만 사용하고 범위 탐색이 전혀 없을 때
- 단, 실무에서는 나중에 범위 탐색이 필요해지는 경우가 많아 처음부터 B+ 트리를 쓰는 것이 안전

**LSM Tree를 선택하는 경우**
- 로그 수집, IoT 데이터, 채팅 메시지 등 **쓰기가 극단적으로 많은** 워크로드
- 읽기보다 쓰기가 많은 NoSQL 환경 (Cassandra, LevelDB, RocksDB)

## 11. SQL 처리기에서 WHERE id = ? 일 때 B+ 트리를 타게 하는 분기 로직

옵티마이저 없이 수동으로 분기하는 방법:

### 핵심 아이디어

```c
void execute_select(ParsedQuery *query, Table *table, BPTree *index) {
    if (query->where_col != NULL &&
        strcmp(query->where_col, "id") == 0 &&
        query->where_op == OP_EQ) {
        
        // B+ 트리 인덱스 탐색
        int target_id = atoi(query->where_val);
        void *record = bptree_search(index, target_id);
        print_record(record);

    } else {
        // 인덱스 없음 → 선형 탐색
        for (int i = 0; i < table->row_count; i++) {
            if (matches(table->rows[i], query)) {
                print_record(&table->rows[i]);
            }
        }
    }
}
```

### 분기 조건 정리

| 조건 | 탐색 방법 |
|---|---|
| `WHERE id = 숫자` | B+ 트리 단일 탐색 |
| `WHERE id BETWEEN 숫자 AND 숫자` | B+ 트리 범위 탐색 (리프 연결 리스트 활용) |
| `WHERE id > 숫자` | B+ 트리 범위 탐색 |
| `WHERE name = '...'` | 선형 탐색 |
| `WHERE score > 숫자` | 선형 탐색 |
| `WHERE` 절 없음 | 전체 선형 탐색 |

### SQL 파싱 시 확인해야 할 것

```
1. WHERE 절이 있는가?
2. WHERE 절의 컬럼이 'id' 인가?
3. 연산자가 =, >, <, BETWEEN 중 무엇인가?
→ 위 세 가지를 확인해서 분기
```
