# B+ 트리는 이름으로도 찾을 수 있을까?

## 결론부터

`가능합니다.`

하지만 중요한 것은
`무엇을 기준으로 인덱스를 만들어 두었는가` 입니다.

## 아주 쉽게 설명하면

B+ 트리는 `번호만 찾는 기술`이 아닙니다.

더 정확히 말하면,
`미리 정해둔 기준으로 데이터를 빠르게 찾는 방법`입니다.

예를 들어:

- `id` 기준으로 정리하면 `id 검색`이 빠릅니다.
- `name` 기준으로 정리하면 `name 검색`이 빠릅니다.
- `score` 기준으로 정리하면 `score 검색`이 빠릅니다.

즉, 무엇으로 정리했느냐에 따라 빨라지는 검색이 달라집니다.

## 도서관으로 다시 생각해 보면

도서관에서 책을 찾을 때 꼭 번호만 쓰는 것은 아닙니다.

- 책장 자체는 등록번호 순으로 정리될 수 있고
- 컴퓨터 검색대에서는 제목 순으로 찾을 수 있고
- 저자 이름으로도 찾을 수 있습니다

이 말은,
`찾는 기준마다 따로 정리된 색인표가 있을 수 있다`는 뜻입니다.

## 우리 프로젝트에 연결하면

이번 과제에서는 보통 `id`를 기준으로 B+ 트리를 만듭니다.

그 이유는:

- `id`는 보통 중복되지 않습니다
- 숫자라서 비교가 쉽습니다
- `WHERE id = ?`처럼 검색 조건이 명확합니다
- 인덱스 효과를 보여주기 가장 쉽습니다

## 이름으로 찾으려면?

이름으로도 빨리 찾고 싶다면
`name`을 기준으로 한 인덱스를 따로 만들어야 합니다.

예를 들면:

- `id` 인덱스: 플레이어 번호로 찾기
- `name` 인덱스: 플레이어 이름으로 찾기

이렇게 여러 인덱스를 둘 수도 있습니다.

## 그런데 왜 이름은 과제에서 덜 쓰일까?

이름 검색은 `id`보다 조금 더 복잡할 수 있습니다.

이유는:

- 같은 이름이 여러 명 있을 수 있습니다
- 문자열 비교는 숫자 비교보다 복잡합니다
- 부분 검색까지 들어가면 더 어려워집니다

예를 들어:

- `name = 'ShadowBlade'` 는 비교적 단순합니다
- `name LIKE 'Shadow%'` 는 더 복잡합니다
- `name LIKE '%Blade%'` 는 더 복잡합니다

그래서 이번 과제에서는 보통
`id 인덱스`를 먼저 구현하고,
다른 필드는 선형 탐색과 비교하는 방식이 가장 자연스럽습니다.

## 한 문장 정리

`B+ 트리는 번호만 찾는 구조가 아니라, 미리 정해둔 기준값으로 빠르게 찾게 해주는 구조다.`
