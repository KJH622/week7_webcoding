# 트리 높이와 ORDER의 장단점 정리

## 먼저 알아둘 점

트리 높이는 보통 `직접 설정하는 값`이라기보다,
아래 요소들에 의해 `결과적으로 결정되는 값`입니다.

- 데이터 개수
- `ORDER` 값
- 노드가 얼마나 차 있는지

즉, 실제로 우리가 주로 정하는 것은 `ORDER`이고,
트리 높이는 그 결과로 달라집니다.

## ORDER가 무엇인가?

이번 프로젝트에서는 `ORDER`가
`한 노드에 키와 자식을 얼마나 담을 수 있는가`
를 정하는 기준입니다.

예를 들어 `BP_ORDER = 4`라면:

- 키는 최대 4개
- 자식 가지는 최대 5개

가질 수 있습니다.

## ORDER가 작을 때

### 장점

- 구조를 이해하기 쉽습니다
- split이 자주 일어나서 동작 원리를 보기 좋습니다
- 디버깅이 쉽습니다
- 발표할 때 설명하기 쉽습니다

### 단점

- 한 노드에 적게 담기므로 트리 높이가 빨리 커질 수 있습니다
- 탐색할 때 더 많은 단계로 내려가야 할 수 있습니다
- split이 자주 발생합니다
- 데이터가 많아질수록 성능상 불리할 수 있습니다

## ORDER가 클 때

### 장점

- 한 노드에 많은 키를 담을 수 있습니다
- 트리 높이가 낮게 유지되기 쉽습니다
- 탐색 시 내려가는 단계 수가 줄어듭니다
- split이 덜 발생합니다
- 대용량 데이터에서 보통 더 유리합니다

### 단점

- 한 노드 안에서 비교해야 할 키 수가 많아집니다
- 삽입, 이동, split 구현이 더 복잡해질 수 있습니다
- 노드가 너무 크면 메모리 낭비가 생길 수 있습니다
- CPU 캐시 효율이 나빠질 수도 있습니다
- 과제나 발표에서는 동작을 눈으로 설명하기 어려워질 수 있습니다

## 트리 높이가 낮을 때

### 장점

- 루트에서 리프까지 적은 단계만 거칩니다
- 검색이 빠를 가능성이 큽니다
- 디스크 기반 환경에서는 접근 횟수를 줄이기 좋습니다

### 단점

- 높이를 낮추기 위해 노드를 너무 크게 잡으면
  한 노드 내부 비교 비용이 커질 수 있습니다
- 노드 크기가 커지면 구현과 관리가 복잡해질 수 있습니다

## 트리 높이가 높을 때

### 장점

- 각 노드 크기를 작게 유지할 수 있습니다
- 교육용 구현에서는 구조를 따라가기 쉬울 수 있습니다

### 단점

- 탐색할 때 거치는 노드 수가 많아집니다
- 삽입 시 split 전파가 더 자주 일어날 수 있습니다
- 대용량 데이터에서는 비효율적일 수 있습니다

## 왜 실제 DBMS는 큰 ORDER를 자주 사용할까?

실제 DBMS는 보통
`한 노드에 가능한 많은 키를 담아서 트리 높이를 낮추는 방향`
을 선호합니다.

이유는 디스크나 페이지 단위 접근 비용이 크기 때문입니다.
한 번 읽을 때 더 많은 정보를 담으면
전체 탐색 단계 수를 줄이기 좋습니다.

## 우리 프로젝트에서는 왜 작은 ORDER를 쓸 수 있을까?

이번 과제는 실무 최적화보다
`구현`, `검증`, `설명`
이 중요합니다.

그래서 작은 ORDER는 다음 장점이 있습니다.

- split이 빨리 보여서 구현 확인이 쉬움
- 테스트가 쉬움
- 발표 때 구조 설명이 쉬움

즉, 과제용으로는 작은 ORDER가 이해하기 좋고,
실무용으로는 더 큰 ORDER가 성능상 유리한 경우가 많습니다.

## 한 문장 정리

`ORDER가 작으면 이해와 구현은 쉽지만 트리 높이가 커지기 쉽고, ORDER가 크면 트리 높이는 낮아지기 쉽지만 노드 내부 비교와 구현 복잡도가 커질 수 있다.`
